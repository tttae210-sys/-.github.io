# [Cloud] 1주차: 클라우드 컴퓨팅 정의와 AWS 핵심 구조 정리

안녕하세요! 오늘은 현대 IT 인프라의 근간이 되는 클라우드 컴퓨팅의 정의와 AWS의 글로벌 인프라 구조에 대해 정리해보려 합니다. 이제는 선택이 아닌 필수가 된 클라우드 과연 어떤 원리로 돌아갈까요?

---

## 1-1. 클라우드 컴퓨팅이란 무엇인가?
클라우드 컴퓨팅이란 **인터넷을 통해 IT 자원(컴퓨팅, 서버, 스토리지, 앱 등)을 온디맨드로 제공하는 모델**입니다.

**주요 특징**: IT 리소스를 직접 구매해서 관리할 필요 없이, 필요한 만큼 빌려 쓰고 사용한 만큼만 비용을 지불합니다.
**NIST 정의**: 필요에 따라 언제 어디서나 편리하게 공유 풀에 접근하여 컴퓨팅 자원을 사용할 수 있도록 하는 서비스 형태를 의미합니다.

---

## 1-2. 온프레미스 vs 클라우드
전통적인 방식과 클라우드의 가장 큰 차이는 누가 인프라를 소유하고 관리하느냐에 있습니다.

| 항목 | 온프레미스 | 클라우드 |
| :--- | :--- | :--- |
| **인프라 관리** | 기업이 직접 구축 및 관리 클라우드 공급자가 인프라 제공/관리 |
| **초기 비용** | 높은 자본 지출 낮은 초기 비용 사용량 기반  |
| **확장성** | 하드웨어 증설 필요 (느림) 필요 시 자동 확장 가능 (빠름) |
| **가용성** | 장애 대비 수동 이중화 필요 다중 가용 영역 기반 고가용성 |

## 1-3. 클라우드 도입의 기대 효과
클라우드를 도입하면 비즈니스 측면에서 다음과 같은 핵심 이점을 얻을 수 있습니다.

1. **비용 효율성**: 초기 하드웨어 구매 비용 없이 사용량 기반 요금으로 IT 예산을 최적화할 수 있습니다.
2. **확장성 및 탄력성**: 트래픽 급증 시 리소스를 즉각 확장하고, 필요 없을 땐 축소하는 '탄력성'을 제공합니다.
3. **속도 및 민첩성**: 인프라 준비 시간 없이 앱과 서비스를 신속하게 배포하여 시장 대응 속도를 높입니다.
4. **글로벌 접근성**: 전 세계 리전에 인프라를 배포하여 글로벌 사용자에게 빠른 서비스를 제공합니다.

---

## 1-4. AWS(Amazon Web Services)의 탄생
AWS는 아마존 내부의 복잡한 IT 인프라를 표준화하려는 시도에서 시작되었습니다.

* **성장 배경**: 제프 베조스는 2002년 사내 메일을 통해 모든 팀이 서비스 인터페이스를 통해서만 데이터를 공개하고 통신하도록 강제했습니다.
* **상업화**: 이 내부 플랫폼의 표준화 경험을 바탕으로 2006년 EC2와 S3를 시작으로 퍼블릭 클라우드 시장을 개척했습니다.

## 1-5. AWS 글로벌 인프라 구조
AWS는 전 세계적으로 분산된 데이터 센터 네트워크를 통해 안정성을 보장합니다.

**리전 (Region)**: 지리적으로 분리된 위치로, 최소 3개 이상의 가용 영역으로 구성됩니다.
**가용 영역 (Availability Zone, AZ)**: 고가용성과 장애 격리를 위해 설계된 논리적 데이터 센터 그룹입니다.
**엣지 로케이션 (Edge Location)**: 사용자 가까이에서 콘텐츠를 캐싱하여 제공하는 시설물입니다.

---

## 1-6. 리전 선택 시 고려해야 할 4가지 요소
서비스를 배포할 리전을 선택할 때는 비즈니스 요구사항에 따라 전략적인 결정이 필요합니다.

1. **지연 시간**: 사용자와 가까울수록 속도가 빠릅니다.
2. **서비스 가용성**: 모든 리전이 동일한 서비스를 지원하지 않으므로 확인이 필요합니다.
3. **비용**: 같은 서비스라도 리전에 따라 요금이 다를 수 있습니다.
4. **법률 및 데이터 주권**: 금융, 의료 등 특정 데이터를 국가 내에 보관해야 하는 법적 요구사항을 준수해야 합니다.

### 💡 1주차 학습 회고
그동안 막연하게 가상 서버라고만 생각했던 클라우드가 실제로는 비즈니스의 CapEx(자본 지출)를 OpEx(운영 비용)로 전환하여 민첩성을 극대화하는 전략적 도구라는 점을 배웠습니다. 
특히 AWS의 탄생 배경이 아마존 내부의 소통 방식에서 시작되었다는 점이 매우 흥미로웠습니다.
